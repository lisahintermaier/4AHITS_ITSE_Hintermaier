## Übung (ssh auf Metasploitable)      
### Aufgabenstellung:        
Als Vorbereitung für die folgende Übung.        

Konfiguriere passwortloses ssh Login auf Metasploitable mit Hilfe eines ssh Public/Private keys.       

Betrachte besonders die notwendigen Änderungen in ~/.ssh/authorized_keys, diese Datei muss in der nachfolgenden Übung manipuliert werden.           

Hinweis: der ssh-server auf Metasploitable unterstützt nur Schlüssel im RSA Format (nicht die neueren default Formate wie ed255…).
Der Kali ssh Client lehnt die Verbindung mit diesem alten Schlüssel standardmäßig ab, aber man kann es mit Optionen wieder erzwingen.
Bei ssh Verbindungsproblemen hilft häufig die Verwendung der Verbose Optionen des ssh clients (-v, -vv oder -vvv).

### Lösung:     
### RSA - Key auf Kali erzeugen      
Befehl: ```$ ssh-keygen -t rsa -b 2048 -m PEM -f ~/.ssh/metasploitable_rsa ```       
![](https://github.com/user-attachments/assets/cae98428-14cd-4564-9391-5b432406496d)      
Erklärung:     
-t rsa → RSA erzwingen       
-b 2048 → 2048 Bit für die alten Server       
-m PEM → vermeidet neue OpenSSH-Formate        
-f → eigener Dateiname      
 
### Public Key anzeigen lassen      
Befehl: ```$ cat ~/.ssh/metasploitable_rsa.pub```       
![](https://github.com/user-attachments/assets/6db15bb3-bcf8-49bf-99d4-09ff70a91275)   

### per Passwort von Kali auf Metasploitable einloggen:      
Befehl: ```$ ssh msfadmin@192.168.18.51```     
Info: dieser Befehl funktioniert nicht, weil der SSH- Server alte Algorithmen benutzt.      
-> Fehlerbehebung:      
Befehl:      

```sh
$ ssh -o HostKeyAlgorithms=+ssh-rsa \
-o PubkeyAcceptedAlgorithms=+ssh-rsa \
msfadmin@192.168.18.51
```
        
![](https://github.com/user-attachments/assets/d8a9813a-20e0-4bb4-97d6-0d997ead9086)        
- Auf dem Foto kann man erkennen, dass man nun im Kali auf Metasploitable eingeloggt ist.       

### ssh Verzeichnis im Metasploitable anlegen
Befehle: 

```sh
mkdir -p ~/.ssh
chmod 700 ~/.ssh
```

### authorized_keys erstellen:     
```nano ~/.ssh/authorized_keys```

-> key an diese Datei anhängen:       
Befehle:  

```sh
$ cat ~/.ssh/metasploitable_rsa.pub | ssh \
-o KexAlgorithms=+diffie-hellman-group1-sha1 \
-o HostKeyAlgorithms=+ssh-rsa \
msfadmin@192.168.18.51 "cat >> ~/.ssh/authorized_keys"
```

### Prüfen, ob man sich ohne Passwort einloggen kann:   
Befehle:    
```sh
 ssh -i ~/.ssh/metasploitable_rsa \
-o KexAlgorithms=+diffie-hellman-group1-sha1 \
-o HostKeyAlgorithms=+ssh-rsa \
-o PubkeyAcceptedKeyTypes=+ssh-rsa \
msfadmin@192.168.18.51
```
Erklärung: 
-i ~/.ssh/metasploitable_rsa      
→ sagt SSH, welchen Private Key es benutzen soll.       

-o PubkeyAcceptedKeyTypes=+ssh-rsa       
→ akzeptiert alte RSA-Keys, die der Metasploitable SSH-Server verlangt.     
Neuere SSH-Clients lehnen die standardmäßig ab.      

-o HostKeyAlgorithms=+ssh-rsa    
→ sagt dem Client, dass auch der alte HostKey-Algorithmus erlaubt ist.      
Ohne das kommt Fehlermeldung no matching host key type kommt.      

![](https://github.com/user-attachments/assets/da087b83-a062-4e42-8eed-334ce85b0e58)             
-> Funktioniert!     


## Übung (portmapper/nfs)
### Aufgabenstellung: 
Metasploitable 2 enthält eine Schwachstelle im Zusammenhang mit portmapper/nfs.

- Informiere dich grob über nfs (network file system) und den Zusammenhang mit portmapper
- Was kannst du über nmap und nmap scripts über die nfs Schwachstelle herausfinden?
- Nutze die Schwachstelle aus indem du über das File-System einen ssh key in das Homeverzeichnis von root hineinschwindelst. Das Ziel ist, dass vom Angreifer-System aus ein passwortloses SSH login auf root@metasploitable möglich ist.

### Lösung: 

### nfs im Zusammenhang mit portmapper: 
Das Network File System (NFS) ist ein Netzwerkprotokoll zur Bereitstellung von Dateisystemen über IP-Netzwerke. Es ermöglicht Clients, entfernte Verzeichnisse so einzubinden, als wären sie lokal gespeichert.

Ältere Versionen (NFSv2/v3) basieren auf dem RPC-Mechanismus und verwenden den Dienst rpcbind (früher Portmapper). Dieser läuft standardmäßig auf Port 111 (TCP/UDP) und teilt Clients mit, auf welchen Ports die jeweiligen NFS- und RPC-Dienste erreichbar sind. NFS selbst verwendet in der Regel Port 2049.
   
Fehlkonfigurationen stellen ein erhebliches Risiko dar, insbesondere wenn:
- Port 111 (rpcbind) öffentlich erreichbar ist
- NFS-Shares ohne Einschränkung exportiert sind
- die Option no_root_squash aktiviert ist
- keine Netzwerkeinschränkungen bestehen
-> Da NFSv2/v3 häufig nur auf IP-basierter Vertrauensprüfung beruhen, können Angreifer durch Enumeration sensible Informationen erhalten oder im schlimmsten Fall Root-Zugriff erlangen.

### Nmap und NFS-Schwachstellen

Mit dem Netzwerkscanner Nmap lassen sich offene RPC- und NFS-Dienste identifizieren.

Relevante Funktionen:
- Portscan auf 111 und 2049
- Service-Erkennung (-sV)
- NSE-Skripte wie rpcinfo und nfs-showmount zur Enumeration exportierter Freigaben

-> Über diese Skripte können exportierte Verzeichnisse, erlaubte Hosts und RPC-Dienste ausgelesen werden. Dies ermöglicht eine Bewertung möglicher Fehlkonfigurationen.
